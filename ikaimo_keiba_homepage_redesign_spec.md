# いかいもAI競馬 首页改版方案（当前项目可执行版）

## 1. 文档目的

本文档用于把原始首页改版设想收敛为**适配当前项目现状**的实施规范。

当前项目并不是 SSR 框架，而是：

- FastAPI 返回公开站点入口 HTML
- 前端使用 React SPA 渲染首页
- 首页主要数据来自 `/keiba/api/public/board`

因此，本版方案的目标不是一次性重做整套渲染架构，而是在**保留现有数据看板能力**的前提下，把首页升级为：

- 内容型首页
- 数据入口页
- 对搜索和审核更友好的说明型首页

---

## 2. 当前项目现状

### 2.1 当前公开首页架构

- 路由入口由 FastAPI 提供
- `/keiba` 返回前端构建后的 `index.html`
- React 在客户端加载后再请求 `/api/public/board`
- 首页主体目前以“今日预测 / 回收率 / 赛程卡片”为核心

### 2.2 当前已存在的可复用能力

当前后端 `board` payload 已经提供了不少首页改版需要的数据：

- `target_date_label`
- `generated_at_label`
- `totals`
- `summary_cards`
- `hero.lead_race`
- `races`
- `daily_predictor`
- `history`

当前前端也已有可复用模块：

- 站点页头
- 左侧导航
- 首页统计卡
- 今日赛程看板
- 履历页
- 静态说明页

### 2.3 当前缺失能力

以下能力目前项目中不存在或不足：

- 真正的 SSR 页面级渲染链路
- 独立的文章系统
- 深度分析文章详情页体系
- 可人工维护的首页导读数据
- 首屏 HTML 中直接输出的首页正文内容

---

## 3. 本次改版的实际目标

本次改版的核心目标仍然保持不变，但落地方式需要调整。

首页首屏要在尽量短的阅读时间内回答用户以下问题：

1. 这个站是做什么的
2. 这个站和普通 AI 赛马推荐站有什么区别
3. 这个站的分析流程是什么
4. 用户现在应该先看哪里

本次改版必须达到以下结果：

- 首页第一印象从“工具页”调整为“分析型媒体首页”
- 保留现有赛程、模型、回收率、结果等数据能力
- 首屏加入清晰的说明文本，而不是只有卡片和数字
- 增加“人类导读感”而不是纯模型输出感
- 增加方法说明和内容入口
- 让首页更适合被理解为有原创整理价值的分析站

---

## 4. 需要修正的原始要求

## 4.1 关于 SSR / SSG

原始方案中要求：

- 首屏核心内容必须 SSR / SSG

这个要求在方向上是对的，但对当前项目来说过重。

### 修正后的要求

首页首屏核心文案需要满足以下条件：

- 由服务端返回的 `/keiba` HTML 中直接包含首屏核心说明文本
- 不要求整站迁移为完整 SSR
- 数据看板部分仍允许保留客户端请求 `/api/public/board` 的方式

### 结论

本次改版不做整站 SSR 重构，只做：

- 首页首屏核心文案服务端注入
- 数据看板继续走现有 SPA 流程

这能显著降低改造成本，同时满足搜索与审核对首屏内容可见性的要求。

## 4.2 关于“最新深度分析”

原始方案中要求首页展示真正的分析文章入口，这个方向保留。

但当前项目没有现成文章系统，所以不能把一期目标写成：

- 每日生产多篇深度文章
- 立刻具备完整文章 CMS

### 修正后的要求

一期先实现“精选内容入口”：

- 可先用静态内容页或 JSON 驱动内容卡片承载
- 内容类型优先为：
  - 分析方法
  - 首页导读说明
  - 预测阅读指南
  - 复盘方法说明

二期再扩展为真正的：

- 赛前分析页
- 分歧分析页
- 赛后复盘页

## 5. 首页新的信息架构

建议首页改为以下顺序：

1. Hero 首屏
2. 本日の見どころ
3. 予想の作り方
4. 注目コンテンツ
5. 今日のレース看板
6. モデル実績・回収率摘要
7. 新手说明 / FAQ 导航
8. 页脚

说明：

- 原有“赛程、印、买い目、回収率”等模块保留
- 但它们不再占据首页首屏最上方
- 首页上半段先讲清楚“价值”和“看法”
- 首页下半段再承接“数据入口”

## 7. 首页模块实施规范

## 7.1 Hero 区块

### 目标

让用户一眼理解本网站不是单纯的：

- LLM 推荐站
- 只报买い目的工具站
- 只展示模型排序的结果页

而是：

- 独自量化模型筛选
- LLM 输出马券策略
- 统一公开、比较、验证、复盘

### 内容结构

- Eyebrow
- H1
- 100 到 180 字左右的说明文
- 2 个 CTA
- 右侧今日摘要卡

### 右侧摘要卡数据优先来源

优先复用现有 `board` 数据：

- `totals.race_count`
- `generated_at_label`
- `hero.lead_race`
- `summary_cards`

### 文案原则

- 说明性优先
- 不用夸张赌徒风文案
- 强调“比较、验证、复盘、方法”

---

## 7.2 本日の見どころ

### 目标

这是首页最关键的“编辑感”模块。

即使当前没有完整人工编辑后台，也必须让首页看起来是“站点在组织内容”，而不是“API 原样倾倒结果”。

### 一期实现方式

采用“自动生成 + 固定模板文案”的方式实现：

- 导语文本由站点统一拼装
- 不直接复用模型原始 comment 文本
- 三张卡片从现有数据中推导

### 三张卡片建议

- `注目`
  - 取当日主推或 lead race
- `分歧`
  - 取模型判断差异较大的场次
- `堅実`
  - 取一致度高或信息结构较稳定的场次

### 后续增强方式

后续允许增加按日期维护的覆盖数据：

- 有覆盖时优先使用人工导读
- 没有覆盖时回退到自动生成

---

## 7.3 予想の作り方

### 目标

快速说明本站不是“AI 随便猜”，而是有完整流程的分析站。

### 固定结构

使用 3 个 step 卡片：

1. 独自量化モデルで有力馬を抽出
2. LLMで買い目構成を比較
3. 結果と回収率を継続検証

### 数据要求

这一块不依赖动态数据，使用固定说明文即可。

### 补充要求

需要有明显链接跳转到：

- 分析方法页
- 指南页

---

## 7.4 注目コンテンツ

### 目标

给首页一个“可阅读入口”，避免首页只有赛程和统计。

### 一期实施方式

先不强依赖真实文章系统，采用以下两种方式之一：

1. 静态内容页卡片
2. 本地 JSON / 前端常量驱动的内容卡片

### 一期推荐内容类型

- 分析方法
- 预测阅读指南
- 回收率怎么看

### 二期扩展内容类型

- 赛前深度分析
- 模型分歧分析
- 赛后复盘
- 方法论长文

## 8. 技术实现原则

## 8.1 首页不做整站重构

本次不将首页改造成新的前后端架构，而是在当前结构中增量改造。

### 保留

- FastAPI 公开站点路由
- React SPA 页面主体
- `/api/public/board` 数据接口

### 新增

- 服务端注入首页首屏核心文本
- 新的首页 section 组件
- 首页内容配置结构

## 8.2 首屏 HTML 可见策略

必须保证以下内容直接出现在返回给浏览器的 HTML 中：

- Hero H1
- Hero 说明文
- 本日の見どころ导语
- 方法摘要短文
- 注目コンテンツ的标题与摘要

### 推荐方式

在 FastAPI 返回首页 `index.html` 时，对首页路径额外注入一段首屏静态内容容器。

这部分内容：

- 可作为 SEO / 审核可见正文
- 也可在前端 hydration 后继续保持一致展示

## 8.3 动态数据来源

动态内容继续来自 `build_public_board_payload()`。

一期不建议拆新的复杂接口。

如果需要，可在现有 `board` payload 上新增：

- `homepage_content`
- `editorial_guide`
- `featured_content`
- `hero_summary`

## 11. 页面与内容策略

## 11.1 一期优先复用现有静态页

当前已有静态说明页能力，因此一期应优先增强这些页面，而不是新建复杂系统：

- `/keiba/about`
- `/keiba/guide`
- `/keiba/methodology`

### 这三页应承担的职责

- `about`
  - 站点定位
  - AI 模型与量化模型的关系
- `guide`
  - 如何阅读预测页
  - 如何理解印与买い目
- `methodology`
  - 分析流程
  - 回测 / 结果 / 免责说明

## 11.2 二期再扩展文章页

二期如需做真正内容站，可新增：

- `/keiba/insights/{slug}`
- `/keiba/reviews/{slug}`
- `/keiba/method/{slug}`

一期不强制要求实现这些路由。

## 12. 文案与语气规范

首页整体语气调整为：

- 分析
- 比較
- 検証
- 振り返り
- 条件適性
- リスク判断
- 確率とオッズのバランス

避免使用：

- 爆益
- 必勝
- 神予想
- 稼げる
- 跟单感太强的表达

所有用户可见文案统一使用日语。

---

## 13. 视觉方向规范

首页视觉方向应更接近：

- 分析媒体首页
- 研究报告入口页
- 编辑型数据站

而不是：

- 纯赌徒风工具页
- 高刺激竞价页
- 只有卡片堆叠的结果墙

### 设计原则

- 首页上半部强调文字可读性
- 数据感保留，但不压过正文
- 首屏 CTA 清晰
- 移动端优先保证阅读顺序

### 不建议本次做的事

- 全站视觉语言大翻修
- 首页与详情页完全脱离现有样式体系
- 只做视觉，不补文本结构

---

## 14. 验收标准

改完后，首页应满足以下标准：

- 用户在不滚动时能理解站点价值主张
- 首屏存在真实说明文本，而不是只有数字和卡片
- 首页出现“本日の見どころ”形式的导读模块
- 首页出现“予想の作り方”摘要
- 首页出现“注目コンテンツ”入口
- 原有赛程和模型看板仍可访问
- 原有数据能力未被破坏
- 首页服务端返回 HTML 中可直接看到核心文案

## 16. 给 Codex 的后续实施说明

后续真正开始改代码时，应按以下原则执行：

- 优先复用现有前端组件和数据结构
- 优先在现有 `board` payload 上扩展字段
- 不在一期引入新的重量级内容系统
- 不把任务扩展成整站 SSR 改造
- 必须同时改“信息架构 + 文本结构 + 首页顺序”，不能只改样式

如果当前实现与本文档冲突，以“当前项目可落地、低风险、可逐步增强”为最高优先级。
